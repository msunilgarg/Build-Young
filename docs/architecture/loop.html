Build Young — Agent Harness (the loop)

You write a spec → it becomes a task; then the loop drains it one at a time — implement → self-check → verify → risk gate → ship → record → next. Everything is committed in the repo (a fresh container resumes); a portable engineering playbook is the core; always-on machinery guards every step. Click any section to drill in.

① You author the work — You → 📐 SPECS/ → 🗂 TASKS.md · the whole process starts here
  • You are the originator human — you write a one-page spec, then promote it to a task.
  • The spec-first gate: non-trivial work → write a 📐 spec first; a bug-level fix → straight to a task.
  • A spec (SPECS/NNN-name.md) pins what / why / behavior / acceptance / out-of-scope / risks — one page, decisions settled with a human.
  • It's promoted into TASKS.md as task(s) that reference it. BACKLOG.md is the someday parking lot.
  • All committed in the repo state, so the loop — and a fresh container — resume from it.
② Kick off a run — ONE on-ramp: run /run-loop locally, or a labeled GitHub issue
  • You — run /run-loop locally human — your Claude subscription; drains the TASKS.md backlog.
  • GitHub issue + “loop-task” label tool → Actions (run-loop.yml). A bug-level issue IS the task; a non-trivial issue is specced first.
  • Use one OR the other for a given task, never both. Same procedure once started.
  • Then the SessionStart hook (every session): resync to origin/main · reinstall the commit guards. — it's a per-session mechanic, not itself an on-ramp.
③ The loop — pick → self-check → verify → risk gate → ship → record → next
Main agent — driver + doer agent · premium
Picks the first unchecked task and reads its 📐 spec, then writes the smallest change that meets the acceptance criteria. Never guesses the next step — it comes from a signal (a failing test, the verifier's gap report, the next backlog item). Stays in the task's file lane.
Self-check — gate your own work first
npm run build · npx vitest run · repo guards (no internal model-id, no resurrected money-sim markers, diagram currency). Fix until green.
Verifier — fresh, independent sub-agent agent · sonnet
Own context, inherits nothing — the doer can't grade its own homework. Re-runs build/tests and grades the diff against the acceptance criteria and the standing rules it is told to read. Returns PASS or FAIL + gaps; fail → fix → re-verify (~3 rounds, then stop and surface the blocker). Runs on the cheaper Sonnet tier — cost discipline, rigor unchanged.
Risk gate — reads the task's risk:
Everything is implemented; only the merge decision differs. In this project it's FULL AUTO — any task auto-merges once the verifier PASSes, regardless of risk (see Risk gate & hard floor below).
Ship — on GitHub
commit → push dev branch → open PR → verify the file diff is non-empty → CI → squash-merge → main → sync.
Record — mark [x] in TASKS.md → next task
Progress lives in committed files, so an ephemeral container resumes exactly where it stopped.
📁 The repo — single source of truth — everything committed, so a fresh container resumes
  • SPECS/ · TASKS.md · .github/workflows · src/ (the app) — all versioned together.
  • The standing rules live here too (next section), so they travel with the code — versioned, reviewable.
  • This is what makes the loop resumable: state survives container resets.
📋 The rules — the verifier grades every change against these
🌐 ENGINEERING-PLAYBOOK.md — THE CORE (portable, any repo)
  • Modularity — one feature = one file · foundation vs features · single source of truth · thin, data-driven router.
  • Parallel agents — 4 anti-collision rules: disjoint files · freeze the foundation · contract-first · merge one-at-a-time.
  • Docs as code — CLAUDE.md is the coordination layer · fix stale docs in the same change · every patterned bug becomes a standing rule.
  • Shipping — small verifiable increments · never push to main · build + test + guards before every commit · enforce via hooks/CI.
  • Loop engineering (§9) — goal + acceptance criteria · doer ⇄ verifier split (fresh context) · model tiering (premium doer · Sonnet verifier) · risk drives autonomy · spec-first gate is trigger-independent · all state committed.
  • Living architecture (§3) — the architecture is a committed, auto-regenerated doc with a currency guard (CI blocks a stale one).
📦 CLAUDE.md — THIS repo's law (project-specific)
House style (optimize-for-less-scrolling · no flag/emoji glyphs in data · statistics integrity) · the module/endpoint map · quality bars (WCAG-AA · ~120 KB JS budget · navLock · act() on clickables). @imports the playbook so the doer auto-loads both.
📦 POSITIONING.md — voice & claims (project-specific)
Tagline “raising builders, not consumers” · what we DON'T claim (no income/return promises; build WITH AI, not coding) · the canonical phrasings (e.g. “Talk to us” · the First-year builder prize).
How the fresh verifier “knows” to read these
It inherits none of the doer's context — the spawn prompt tells it: 🌐 playbook always · 📦 CLAUDE on app/UI · 📦 POSITIONING on copy · 📐 the task's SPEC when it references one.
🛡 Risk gate & hard floor — full auto once the verifier PASSes; pause only on a hard floor
  • FULL AUTO: any task auto squash-merges the moment the verifier PASSes — regardless of risk. The independent verifier is the gate, not a human.
  • Pause only on a HARD FLOOR: an irreversible real-world side-effect → confirm first; else keep going.
  • Examples: send a real email · charge/refund real money · delete prod data · or a value only you have.
  • A task's “stop-and-ask” note is downgraded to flag-in-report (surface it in the PR/summary, don't block).
④ Fan-out — default sequential; parallelize only when provably safe
  • Go parallel only if ALL hold: ≥2 tasks on disjoint files · no foundation change mid-flight · no dependency · the contract (seams) is pinnable up front.
  • Then: worktree-isolated sub-agents → merge one-by-one; after each merge the next rebases.
  • Parallelize the work, not the rigor — each parallel agent runs the full loop (its own verify) on its slice.
⚙ Always-on machinery — enforced by hooks & GitHub Actions, not memory
  • On commit / push / PR → guards + CI block: secrets · internal model-ids · stale diagram (hash) · page-height · push-to-main.
  • GitHub MCP = the agent's API to GitHub (PR · issue · merge). git worktrees = fan-out isolation. SessionStart = resync per session.
  • GitHub is the hub: the repo (state) · Issues (an on-ramp) · Actions (run-loop + CI) · Pull Requests (how every change ships).
🌐 Live site — Vercel — auto-deploys on squash-merge to main
  • A verified change (any risk) → squash-merge → deploy. No manual release step.
🧩 Coordination patterns — the standard multi-agent catalog, mapped to our components
  • Classify-And-Act → our risk gate.
  • Fanout-And-Synthesize → fan-out → merge.
  • Adversarial Verification → doer ⇄ verifier.
  • Loop Until Done → the loop itself.
  • Not used (yet): Generate-And-Filter · Tournament — one doer per task; quality comes from the verifier + guards, not candidate diversity.
